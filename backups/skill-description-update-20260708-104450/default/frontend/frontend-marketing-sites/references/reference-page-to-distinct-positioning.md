# Reference Page To Distinct Product Positioning

Use when a user provides a competitor/reference landing page but explicitly says not to clone it.

## Session Pattern

- Reference observed: a hospitality CRM/customer-care article page with clear H1 hierarchy, navy/cyan SaaS styling, CTA/header patterns, and floating support UI.
- User correction: do not follow that product direction; their product is in-room reception/service delivery, not customer management/CRM.
- Correct response: extract only reusable UI mechanics, then rebuild copy/hero/sections around the user's distinct positioning.

## Durable Approach

1. Inspect the reference for layout mechanics only:
   - hierarchy, header/CTA density, card rhythm, typography scale, spacing, support widgets, and visual weight.
2. Translate the user's positioning into the first screen before coding:
   - For VietSage-like hospitality: "digital front desk inside every room", QR concierge, room service, amenities, housekeeping, local help, multilingual support, staff routing.
3. Add an explicit anti-clone section when the market is adjacent:
   - Example: "Not hotel CRM" / "Works beside PMS" / "guest-facing service layer".
4. Prefer synthetic product UI/mockups over unavailable stock images:
   - Room number badge, guest request cards, service categories, response-time/status panel.
5. Verify with both code checks and visual checks:
   - `npm run lint`, `npm run build`, local browser open, screenshot/vision check for crop/overflow and whether positioning is visually clear.

## Pitfalls

- Do not reuse the reference page's business narrative just because the industry matches.
- Do not let reference-page screenshots push the design into a blog/article layout if the user needs a product landing page.
- Do not assume "hotel guest experience" means CRM; clarify whether the surface is pre-stay CRM, in-stay service, post-stay loyalty, or operations.
