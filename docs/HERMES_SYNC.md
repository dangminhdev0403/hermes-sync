# Hermes sync policy

This repository stores only durable Hermes customizations that are safe and useful after a reinstall.

## Tracked

- Custom and patched skills under `skills/`.
- Specialist profile skill copies under `profiles/<profile>/skills/` when they are intentionally synchronized for backend/frontend/dev-ops/tester workers.
- `SOUL.md`, `profile.yaml`, and durable non-secret memory notes.
- Helper scripts such as `scripts/sonarqube-review.sh`.
- This documentation and `.gitignore` policy.

## Ignored / removed from Git

These files are generated by Hermes at install/runtime or contain local secrets and should not be restored blindly from Git:

- Raw runtime config: `config.yaml`, `profiles/config.yaml`, `profiles/*/config.yaml`.
- Secrets and auth stores: `.env`, `profiles/*/.env`, `auth.json`, `auth.lock`, token/key files.
- Runtime databases and evidence: `state.db*`, `kanban.db*`, `verification_evidence.db`, `*.db*`.
- Runtime locks/PIDs/cache/log/session folders.
- Generated skill metadata: `.usage.json`, `.hub/`, `.curator_state`, prompt snapshots.

## Reinstall restore flow

1. Install Hermes normally and run the setup wizard for local machine secrets/config.
2. Clone or pull this repository into `C:/Users/Admin/AppData/Local/hermes`.
3. Keep local `config.yaml` and `.env` from setup; Git intentionally ignores them.
4. Run a smoke test:

```bash
hermes skills list
hermes -p frontend chat -q "Skill load smoke test. Reply OK." --skills skill-frontend -Q
hermes -p tester chat -q "Skill load smoke test. Reply OK." --skills skill-tester -Q
```

If a config value must be shared later, add a redacted `*.example.yaml` or document the setting here instead of committing the live config file.
